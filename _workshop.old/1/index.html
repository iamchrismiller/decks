RequireJS

AMD?

Asynchronous Module Definition

The Asynchronous Module Definition (AMD) API specifies a mechanism for defining modules such that the module and its dependencies can be asynchronously loaded.

AMD is not a library.

AMD is a specification.

define()

The specification defines a single function "define" that is available as a free variable or a global variable.
define(id?, dependencies?, factory);

Using an ID is not recommended unless necessary.

define(dependencies?, factory);

Why AMD?

  • Dependency Management

    Large applications can easily become very unwieldy with manually managed dependencies.

  • Encourages reusable code

    Easily define and use separate modules.

  • Extremely easy to get started with

    Extends the module pattern you already know (and love).

AMD is different

This ain't your mother's JavaScript

  • In your world, everything is unicorns and rainbows

    New code written adhering to AMD is beautiful and made of cotton candy.

  • Other people's code requires care

    A lot of existing modules don't expect to be loaded asynchronously.

    Most other code expects access to existing global variables, or needs to create their own.

  • Everything is usable, but understanding the differences is important

    Libraries that support AMD often still support non-AMD, global access for compatibility.

Rounding Out Our Foundation

How Do You Test?

You could do it one way:

  1. Write some code you know should probably produce a "4" or something similar.

  2. Go to your browser, refresh, verify that you see a "4".

  3. Success? - Move on. Failure? Go back to 1.

Drawbacks to this approach?

Drawbacks.

  • Slow

    You have to manually test every time.

  • Unscalable

    You really can't test anything except what you are currently working on at any given time.

  • A complete waste

    Every second you manually test is a second gone. You can't reuse your efforts.

    Your a programmer. Have some respect.

Automated Testing

Instead of the previous method:

  1. Write a test to verify some code produces a "4".

  2. Write some code you know should probably produce a "4".

  3. Run your test.

  4. Success? - Move on. Failure? Go back to 2.

Benefits?

Benefits.

  • Grow your tests over time

    Ideally, every line of code can eventually be tested automatically.

  • Your tests keep testing

    Other developers can run your tests to determine if they are breaking anything.

  • It forces you to write against your API earlier

    You start using your code well before it is producing the final product.

    This will expose bugs earlier, allow you to question design choices earlier, and keeps your logic in testable chunks.
